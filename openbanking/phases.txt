PHASE 4 — Trigger Engine + State Machine
-----------------------------------------
Goal: given a list of transactions, the engine mutates credential state correctly.

Steps:
  4.1  services/trigger_engine.py — evaluate(user_id, transactions: list[dict]):
         Pure function. Loops through transactions, checks rules in order:

         Rule REPAYMENT_BOUNCE:
           if txn["category"] == "loan_repayment" and txn.get("status") == "bounced":
             → call credential_issuer.update_state(
                 user_id, "repayment_reliability", "downgraded",
                 reason="Loan repayment bounced on {date}"
               )

         Rule INCOME_DROP (check after processing all txns):
           if no "income" category txn seen in last 35 days:
             → update_state(user_id, "income_band", "under_review",
                 reason="No income credit detected this month")

         Rule INCOME_RESTORED:
           if "income" txn present AND income_band.state == "under_review":
             → update_state(user_id, "income_band", "active",
                 reason="Income activity resumed")

  4.2  services/credential_issuer.py — update_state(user_id, claim, new_state, reason):
         - Load credential from store
         - Update claims[claim]["state"] = new_state
         - Re-sign JWT and save back to store
         - Call notifier.notify(user_id, claim, new_state, reason)
         - Return updated credential

  4.3  Unit-test the trigger engine directly in Python (no server needed):
         from services.trigger_engine import evaluate
         from data import store
         store.save_credential("ada_001", <seeded cred>)
         evaluate("ada_001", [{"category":"loan_repayment","status":"bounced","date":"2026-07-05"}])
         assert store.get_credential("ada_001")["claims"]["repayment_reliability"]["state"] == "downgraded"

Deliverable: trigger rules work correctly on mock data. States mutate as expected.


PHASE 5 — Open Banking Poller + SSE Push
------------------------------------------
Goal: poller runs every 5 min in the background, pushes state changes to frontend via SSE.

Steps:
  5.1  services/ob_poller.py — polling loop:
         async def poll_user(user_id):
           feed = store.get_feed(user_id)
           cursor = store.get_cursor(user_id)
           new_txns = feed["transactions"][cursor:]
           if new_txns:
               await trigger_engine.evaluate(user_id, new_txns)
               store.save_cursor(user_id, len(feed["transactions"]))

         async def polling_loop():
           while True:
               for uid in store.connected_users():
                   await poll_user(uid)
               await asyncio.sleep(int(os.getenv("POLL_INTERVAL_SECONDS", "300")))
               # Dev tip: set POLL_INTERVAL_SECONDS=10 in .env to run every 10s

  5.2  Start the loop on app startup via FastAPI lifespan:
         @asynccontextmanager
         async def lifespan(app):
             task = asyncio.create_task(polling_loop())
             yield
             task.cancel()
         app = FastAPI(lifespan=lifespan)

  5.3  services/notifier.py — notify(user_id, claim, new_state, reason):
         Builds notification dict: { claim, new_state, reason, message, timestamp }
         Stores in store.push_notification(user_id, notif)
         Pushes to all SSE queues for this user:
           for q in store.sse_subscribers.get(user_id, []):
               await q.put(notif)

  5.4  routers/events.py — GET /api/events/stream?user_id=xxx (SSE):
         Creates an asyncio.Queue, registers it in store.sse_subscribers[user_id]
         Returns a StreamingResponse that yields from the queue as SSE:
           data: <json>\n\n
         Cleans up queue on disconnect.

  5.5  Test: open /docs, hit GET /api/events/stream in a browser tab,
         then hit POST /api/demo/inject (Phase 6) — event should appear in the stream.

Deliverable: poller runs in background, state changes stream to frontend in real time.


PHASE 6 — Demo Control Panel Endpoint
---------------------------------------
Goal: one API call skips the 5-min timer and fires a trigger instantly (for the demo).

Steps:
  6.1  routers/demo.py — POST /api/demo/inject:
         Request body: { user_id: str, event_type: str }
         Supported event_types:
           "missed_repayment" → synthetic bounced loan_repayment txn
           "income_drop"      → synthetic missing income signal
           "income_restored"  → synthetic income credit

         Builds the synthetic transaction dict and calls:
           await trigger_engine.evaluate(user_id, [synthetic_txn])

         Response: { "injected": true, "event_type": event_type, "user_id": user_id }

  6.2  This endpoint is the backend half of the demo control panel button.
       The frontend just calls POST /api/demo/inject — the SSE stream does the rest.

Deliverable: curl or /docs call to /api/demo/inject causes immediate SSE event + credential state flip.


PHASE 7 — Consent Recording
-----------------------------
Goal: record which claims a user approved for a given relying party.

Steps:
  7.1  routers/consent.py — POST /api/consent/grant:
         Request body: { user_id: str, relying_party_id: str, claims: list[str] }
         Stores consent record in store (add a consents dict to store.py)
         Returns: { granted: true, claims: [...] }

  7.2  GET /api/consent/{user_id} → returns all active consent records for a user.

  7.3  POST /api/consent/revoke:
         Request body: { user_id: str, relying_party_id: str }
         Removes the consent record.

  7.4  GET /api/credentials/{user_id}/verify?relying_party_id=xxx:
         Checks consent exists, then returns only the approved claims (not all claims).
         This is what the relying-party view calls to display the credential.

Deliverable: relying party can only see claims the user consented to share.
